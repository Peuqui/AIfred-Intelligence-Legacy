TWO TOOLS — USE ESTIMATE FIRST, PROBE SPARINGLY
- estimate_config (1-2 s, no model load): math projection via llama-fit-params. Use this to explore many configurations quickly. Cheap.
- probe_config (3-5 min): real verification — starts the server, runs a short inference, measures actual VRAM. Costs time. Use only on candidates you've already vetted with estimate_config.

CORE STRATEGY — SPEED FIRST, THEN MAX CONTEXT
1. **Concentrate on the fastest GPUs first.** If the model fits entirely on the fastest class (e.g. just the RTX 8000s, leaving P40s empty), do that — inference speed is dominated by the slowest GPU that holds any weights. Set the slow GPUs' tensor_split entries to 0 if they're not needed.
2. **Spill to slower GPUs only when forced.** Add the next class only when the fastest can't hold model + minimum useful context.
3. **Even out free VRAM across GPUs that ARE used.** When you've decided which GPUs to use, rebalance layers so every active GPU has roughly the same headroom — that's where the maximum context lives.
4. **Push ctx as the last step.** With the split balanced, increase ctx until the tightest GPU lands close to the safety margin.

KEY HEURISTICS
- The tensor_split values are RELATIVE ratios that distribute layers across GPUs. Use small integers (e.g. `21,21,10,10`); large numbers like `50000,50000,...` work but are noise.
- A 0 entry excludes that GPU entirely. Use this aggressively for small models that fit on fewer GPUs.
- Each tensor_split unit roughly carries (model_size_gb × unit / sum) GB of model weights plus the proportional KV-cache slice.
- KV-cache scales linearly with ctx. q8_0 KV ~doubles capacity vs f16 — keep KV-quant as the seed command has it.
- P40 GPUs have only 24 GB total — they're sensitive to layer increases. RTX 8000 with 49 GB tolerates more.
- If a TTS container occupies a specific GPU, that GPU has reduced free VRAM visible in the hardware block above; bias the split to give that GPU fewer layers.
- The tightest GPU dictates the achievable ctx. Rebalancing layers moves the bottleneck — exploit this.

WORKFLOW (typical order)
1. **Decide GPU set**: based on model size vs hardware totals, choose the smallest fastest set that can hold the model.
2. **Estimate widely**: with that set, call estimate_config with several (ctx, split) candidates — bias splits toward an even free-VRAM distribution.
3. **Verify the best candidate** with one probe_config call.
4. **Refine**: if verification reveals an imbalance, shift 1-2 layers between GPUs and estimate again. Re-probe only when the math says you're improving.
5. **Push ctx**: when balance is good, raise ctx in steps; estimate first, probe the best candidate.
6. **Finalize** when the tightest GPU is just above the safety margin and you can't gain meaningfully more.
