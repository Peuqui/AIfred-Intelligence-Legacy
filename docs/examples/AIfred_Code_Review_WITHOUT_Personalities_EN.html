I have a Python function with 120 lines that: finds files, filters, reads, transforms, saves, and logs. Should I split it into 6 small functions (Single Responsibility) or is that over-engineering? The function is only called in one place.
AIfred (Thesis) R1

Situation Analysis

You have a monolithic function with 120 lines that fulfills multiple tasks: finding files, filtering, reading, transforming, saving, logging.

This function is only called in one place.

Key Questions for the Decision

  1. Will the function likely be extended or modified in the future? → Yes, since it involves complex data processing.
  2. Are the individual steps logically separated? → Yes: Finding, Filtering, Reading, Transforming, Saving, Logging are semantically different operations.
  3. Can individual parts of the function be reused? → Possibly: e.g., "find files" or "logging" in other scripts.
  4. Is the function hard to test or debug? → With 120 lines and multiple state changes, that's probably the case.
  5. Is the function only called once – is that an argument against splitting? → No. Even single calls benefit from clarity, maintainability, and testability.

Conclusion: Not Over-Engineering – Best Practice

Splitting into 6 smaller functions with Single Responsibility is not over-engineering, but rather good software practice, even if the function is only called once.

Socrates (Critique) R1

Progress

AIfred has clearly identified the core problems of the monolithic function and proposed a structured, practical split. The recommendation to split into six specific functions with clear responsibility is consistent with the Single Responsibility principle.

New Critique

Despite the good structure, a systemic gap remains: The dependency between functions is not explicitly handled. If, for example, transform_data depends on read_file, and read_file fails, this is not clearly communicated in the main function. Error handling remains implicit.

Second: The flow logic is not modularized, but remains in the main function. Even if the individual functions are separated, the control of the flow remains in a single, central place.

Alternative Solution: Pipeline Object

Instead of defining functions only as separate, sequential steps, the entire process should be modeled as a Pipeline object:

class DataPipeline:
    def __init__(self, pattern: str, root: str):
        self.pattern = pattern
        self.root = root
        self.logger = Logger()

    def run(self):
        files = self.find_files()
        filtered = self.filter_files(files)
        for file in filtered:
            try:
                data = self.read_file(file)
                transformed = self.transform_data(data)
                self.save_result(transformed, file)
                self.logger.info(f"Processed {file}")
            except Exception as e:
                self.logger.error(f"Failed to process {file}: {e}")

Revised Recommendation: Pipeline-based Architecture

Socrates rightly pointed out that a pure split into individual functions – without explicit handling of flows and error control – limits the reusability of the entire process and creates implicit dependencies.

New Architecture: Pipeline Object with Modular Components

class DataPipeline:
    def __init__(self, pattern: str, root: str, logger=None):
        self.pattern = pattern
        self.root = root
        self.logger = logger or print  # Fallback for debugging

    def run(self):
        files = self.find_files()
        filtered = self.filter_files(files)
        for file in filtered:
            try:
                data = self.read_file(file)
                transformed = self.transform_data(data)
                self.save_result(transformed, file)
                self.logger(f"Processed {file}", "INFO")
            except Exception as e:
                self.logger(f"Failed to process {file}: {e}", "ERROR")

Socrates (Critique) R2

Progress

AIfred has not only accepted the critique but implemented it consistently: The original function structure was not rejected but extended through integration into a DataPipeline class. The central critique about missing flow and error modularization is now addressed.

New Critique

Despite the improvement, a systemic weakness remains: The logger dependency is too flexible and potentially unsafe. The current implementation allows logger=None and uses print as fallback – practical for debugging but dangerous in production environments.

Second: The save_result method saves data directly to a path that is not passed – instead, file is used, which implicitly assumes the file structure is preserved.

Alternative Solution: OutputConfig Class

class OutputConfig:
    def __init__(self, base_dir: str, format: str = "json"):
        self.base_dir = base_dir
        self.format = format

class DataPipeline:
    def __init__(self, pattern: str, root: str, output_config: OutputConfig, logger=None):
        self.pattern = pattern
        self.root = root
        self.output_config = output_config
        self.logger = logger or self._default_logger

    def _get_output_path(self, file) -> str:
        return f"{self.output_config.base_dir}/{file.stem}.{self.output_config.format}"

Pro/Contra

Pro (AIfred's Position):

  • Reusability of the entire flow
  • Consistent flow control without superfluous logic in the main function

Contra (Counterarguments):

  • Missing explicit configuration for output paths and formats
  • Risk through flexible, untyped logger parameters
  • Implicit assumptions about file paths and names
